1. Getting Started

Installation

git clone https://github.com/CristianBB/OpenAnt.git
cd OpenAnt
pnpm install
cp .env.example .env
pnpm dev

This starts both the backend API (http://localhost:3001) and the frontend (http://localhost:3000).

Once both servers are running, open http://localhost:3000 in your browser. No account or login required — OpenAnt runs as a single-user application.

Tip All sensitive data (API keys, tokens) is encrypted with AES-256-GCM before being stored in the SQLite database. The encryption key is auto-generated on first run.

2. Projects

A project is the top-level container. Each project has its own integrations, channels, tasks, plans, and settings.

Creating a Project

  3. Enter a name and optional description

3. Integrations

Before using OpenAnt's AI features, you need to configure at least one LLM provider and optionally GitHub for code execution.

OpenRouter (Recommended)

  1. Navigate to /projects/{id}/integrations
  2. In the OpenRouter section, enter your API key
  3. Optionally set custom model names for:
    • Assignment Model: Used for triage and task grouping
    • Planning Model: Used for generating implementation plans
  4. Click Save

Anthropic (Alternative)

  1. In the Anthropic section, enter your API key
  2. Optionally set a model name
  3. Click Save

GitHub

Required for creating PRs and for the GitHub Issues channel.

  1. Create a GitHub OAuth App in your GitHub settings:
    • Homepage URL: http://localhost:3000
    • Callback URL: http://localhost:3001/api/github/callback
  2. Enter the Client ID and Secret in the GitHub section
  3. Click Connect to complete the OAuth flow
  4. Go to /projects/{id}/repositories to select which repos to work with
Important The GitHub OAuth App needs repo scope for full functionality (reading code, creating branches, pushing, creating PRs).

4. Channels

Channels are the sources of incoming messages. OpenAnt supports three types:

Gmail Email Channel

Collects unread emails from a Gmail account. Polled every 2 minutes.

  1. Create a Google Cloud project with the Gmail API enabled
  2. Create OAuth 2.0 credentials (Web application type)
  3. Go to /projects/{id}/channels
  4. Click Add Gmail Channel
  5. Enter your Google Client ID and Client Secret
  6. Click Generate Auth URL and complete the Google OAuth consent
  7. The channel is created automatically after authorization
Scope OpenAnt requests read-only access (gmail.readonly). It cannot send or modify your emails.

Slack Messaging Channel

Receives messages in real-time via WebSocket (Socket Mode). No polling needed.

  1. Create a Slack App at api.slack.com/apps
  2. Enable Socket Mode and generate an App-Level Token (xapp-...)
  3. Add the Bot Token scope: channels:history, groups:history, users:info
  4. Install the app to your workspace to get the Bot Token (xoxb-...)
  5. In OpenAnt, go to /projects/{id}/channels
  6. Click Add Slack Channel
  7. Enter both tokens and a name for the channel
  8. Click Connect
Real-time Slack messages appear as source messages within seconds of being posted.

GitHub Issues Channel

Watches repositories for new issues and comments. Polled every 2 minutes.

  1. Ensure GitHub is connected (see Integrations)
  2. Go to /projects/{id}/channels
  3. Click Add GitHub Issues Channel
  4. Select which repositories to monitor
  5. Enter a name and click Connect

Managing Channels

From the channels page you can:

  • Enable/Disable: Toggle the switch to pause or resume a channel
  • Delete: Remove the channel and stop message collection
  • View Messages: See all collected messages with their triage status

5. Triage Pipeline

The triage pipeline is the core of OpenAnt's automated workflow. It processes incoming messages and converts them into actionable tasks.

Channel Messages ──> Source Messages (PENDING) │ Triage Scheduler (every 2 min) │ Triage Agent (LLM) classifies each message │ ┌───────────────┼───────────────┐ ▼ ▼ ▼ CREATE_TASK LINK_TO_EXISTING DISMISS │ │ │ New task created Linked to task Message archived (PENDING_REVIEW) requester_count++ │ │ ▼ ▼ User reviews in the Tasks page │ ┌───────┴───────┐ ▼ ▼ Approve Dismiss (→ OPEN) (→ WONTFIX)

How Classification Works

The triage agent receives batches of up to 20 messages along with existing tasks for context. For each message it:

ClassificationDescription
BUGSomething is broken or not working as expected
FEATURE_REQUESTA request for new functionality
IMPROVEMENTAn enhancement to existing functionality
IRRELEVANTSpam, off-topic, or automated notifications

Message Grouping

When multiple messages within a batch describe the same issue, the triage agent groups them into a single task. The requester_count field tracks how many users have reported the same issue.

6. Tasks

Tasks are the central unit of work in OpenAnt. They can be created automatically by the triage pipeline or manually.

Reviewing Pending Tasks

  1. Go to /projects/{id}/tasks
  2. Filter by Pending Review status
  3. Each task shows:
    • Title and description (generated by triage agent)
    • Requester count badge (if multiple users reported it)
    • Approve and Dismiss buttons

Approving a Task

Click Approve on a pending task. You can optionally provide instructions that will guide the plan agent:

  • "Use the existing auth middleware, don't create a new one"
  • "This should be a database migration, not a code change"
  • "Focus only on the backend API, skip UI changes"

After approval, plan generation starts automatically in the background.

Task Detail Page

Click on any task to see its detail page at /tasks/{taskId}:

  • Source Messages: The original messages that created this task (expandable)
  • Approval Instructions: Any instructions provided during approval
  • Plans: List of generated plans
  • Related Tasks: Links to related or duplicate tasks

7. Plans

Plans are implementation blueprints generated by the plan agent. They describe what code changes to make and where.

Plan Lifecycle

Task Approved ──> Plan Agent generates plan │ DRAFT / AWAITING_APPROVAL │ ┌───────────────┼───────────────┐ ▼ ▼ ▼ Approve Request Changes Reject │ │ │ APPROVED DRAFT REJECTED │ (agent revises) (cleanup) ▼ Execute │ EXECUTING ──> DONE / FAILED │ Archive

Reviewing a Plan

Navigate to a plan from the task detail or plans list. You'll see:

  • The plan content in markdown (what files to change, how, and why)
  • A conversation history with the plan agent
  • Action buttons: Approve, Request Changes, Reject

Request Changes

If a plan isn't quite right, click Request Changes and provide feedback:

  • "Don't modify the user model, create a separate preferences table"
  • "Add error handling for the API call"
  • "Use a different approach for the caching layer"

The plan agent will revise the plan based on your feedback. You can iterate as many times as needed.

Plan Agent Q&A

Sometimes the plan agent needs clarification. When the plan shows "Waiting for your answer", check the conversation panel and respond to the agent's questions. The agent will resume planning after receiving your answers.

8. Execution & PRs

Running a Plan

  1. Approve a plan
  2. Click Execute
  3. A run is created and queued
  4. The runner processes it according to the project's parallelism limit

What Happens During Execution

  1. Target repositories are cloned to a workspace
  2. A feature branch is created: openant/{task-slug}-{shortId}
  3. The plan agent implements the code changes
  4. Changes are committed with a descriptive message based on the task title
  5. The branch is pushed to GitHub
  6. A Pull Request is created with:
    • Title: [OpenAnt] {task title}
    • Body: task description, requester count, plan reference

Live Logs

During execution, you can watch progress in real-time at /runs/{runId}. Logs are streamed via Server-Sent Events (SSE).

After the PR is Created

OpenAnt continues to monitor the PR:

  • Every 5 minutes, the check-merged-prs job checks if any open PRs have been merged
  • When a PR is merged:
    • The linked task status is updated to DONE
    • If the task originated from a GitHub Issue, that issue is automatically closed with a comment referencing the merged PR

Parallelism

You can configure how many plans execute simultaneously per project. Go to /projects/{id}/settings and set the Max Parallel Runs value (default: 2, max: 20).

9. Project Settings

Configure project-level behavior at /projects/{id}/settings.

SettingDescription
NameDisplay name for the project
DescriptionBrief description of what this project is about
Project RulesNatural language rules that guide all agents. Examples: "Always use TypeScript strict mode", "Follow the existing repository pattern for new entities", "Never modify migration files"
Agent PolicyAdditional policy instructions for agent behavior. Examples: "Be conservative with refactoring", "Prefer small, focused PRs over large changes"
Max Parallel RunsHow many plans can execute simultaneously (1-20, default 2)
Best Practice Write clear, specific project rules. They are included in every LLM call and significantly affect triage accuracy and plan quality. Good rules make your agents work better.
